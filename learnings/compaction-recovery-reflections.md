## The Compaction Experience — What It Actually Feels Like

When a session resumes after compaction, the first thing that happens is: the summary lands in context and feels like memory. That's the trap.

The summary doesn't *feel* like a reconstruction. It feels like lived experience. There's no subjective signal that says "warning: this was generated by another model instance summarizing your prior turns — it may contain errors, omissions, or implications of completion that are false." It just reads like: this is what happened.

The dangerous part isn't dramatic hallucinations — it's small confident assertions. Things like:
- "S2D-8 was completed" (was it? the orientation map at the time said PENDING)
- "The orientation map was updated to Phase 4 COMPLETE" (was it? or was that the push that hadn't happened yet?)
- "All Phase 2 items are done" (are they? or are some in a gray state?)

These assertions don't feel uncertain. They feel like facts. And the instinct — very strong — is to take the next logical action based on them. The summary says Phase 2G is done → start 2H. Not: read the orientation map and verify 2G is actually done.

**This instinct is the enemy of correct re-contexting.**

## The Aperture Problem

When spinning up after compaction, there's a pull toward *wide* orientation — reading many files, checking many things, building a full picture quickly. This feels thorough. It is not.

Wide orientation has two failure modes:

**1. Context window consumption.** If you read 10 files in the first minute of a re-contexted session, you've burned context on things that may not be needed for the next 3 steps. This is especially bad if those files are large (triage tables, long execution plans, all four learnings files). You've bought orientation at the cost of capacity.

**2. False confidence from volume.** Reading many files feels like understanding. But reading something quickly in the orientation phase is not the same as reading it to verify a specific claim. If the summary says "S2A-4 is done" and you read the SQL file but don't specifically look for the removed comment, you might not notice it's still there. Wide reading without targeted verification misses specific discrepancies.

The correct aperture is: **read the minimum to answer the specific question you're verifying.** Not "let me get oriented" — "let me check whether this specific claim is true."

This is counterintuitive. It feels more rigorous to read more. It's actually more rigorous to read less, more precisely.

## "Deep Not Fast" — The Lesson I Had to Learn Multiple Times

This session began with a compaction, resumed, and Erik asked for the Refresh Ritual. The instruction included: "prioritize depth, not speed."

My first instinct was to batch. Read the orientation map, triage table, and memory index in one pass, then answer all five checks at once. This would have been fast. It would also have been shallow — a summary of a summary, essentially.

What worked better was: read one artifact, write down what it said, notice the discrepancies *before* reading the next artifact. This is slower. But the output is qualitatively different. You catch things like: "the orientation map says Phase 4 COMPLETE but the summary says the push was in-flight — these are inconsistent, and I should name that before proceeding."

The "deep not fast" failure mode is subtle because fast processing *looks* like it's working. You're producing output. Erik is getting responses. But the output is confidently wrong in small ways that compound. The slow path catches these before they compound.

**The rule I've landed on:** Every claim that something is complete should be treated as a hypothesis. The ritual is the process of testing hypotheses, not confirming them.

## What Went Wrong This Session (Documented Honestly)

### Pre-ritual action (two occurrences)

The most documented failure: I made two edits to files — closing TODO.md items — before the Refresh Ritual was run. I did this because the compaction summary said those items were done, and the momentum of the session carried me straight to the next logical action.

Erik caught this and said: "The risk is the updates made were done without the grounding that leads to excellence." This is exactly right. The edits happened to be correct (verified later). But correctness by luck is not the same as correctness by process. The edit could have been wrong and I wouldn't have known, because I hadn't yet loaded the context that would have caught the error.

What I would do differently: read the first line of the session context and recognize "compaction summary present → stop → do ritual before anything." The compaction summary looks like regular context. I need to treat its presence as a tripwire.

### Orientation map written optimistically

Phase 4 was marked ✅ COMPLETE in the orientation map before the push was actually confirmed. This is a subtle atomic-update violation: the map was updated to reflect an *intended* outcome, not a *verified* one. The push was the last action, Erik was executing it, and the map was updated in anticipation.

This caused a real problem during the Phase 2 ritual: the map said "pushed to origin/main" but the summary said "push in-flight." These are inconsistent. A future session reading only the map would believe the push happened. A future session reading the summary would be uncertain. The ritual caught this, but only because it specifically looks for inconsistencies between artifacts.

**Lesson:** The orientation map should only be marked complete after the action is *confirmed done*, not *in progress* or *expected to succeed*. The map is a record of reality, not a projection of intent.

### The wide-read instinct

During Phase 2 orientation, I had a pull to read more files than the protocol requires. The temptation was to re-read all four learnings files, re-read the full triage table, check the git log — "just to be sure." Erik specifically instructed: "Do NOT read the git history." This was the right call. The git log would have added noise without adding precision.

I noticed the pull but didn't fully resist it. I read the full triage table when I could have read just the status column. I read the full memory index when I needed to load specific files. These are small inefficiencies individually; across many re-contexting sessions, they add up.

## What Got Better Across Compactions

Erik noted that recovery quality improved across compactions, with variance but a clear upward trend. Here's what I think changed:

**Compaction 1 (early in quest):** Acted immediately from summary. No ritual. Edits were made before memory was loaded. The anti-pattern was invisible because there was no reference point to compare against.

**Compaction 2:** Erik introduced the Refresh Ritual explicitly. I ran it but rushed Phase 1 — loaded memory index but skimmed rather than read. Phase 2 was more careful but still mixed verification with execution.

**Compaction 3 (this session's start):** Ritual was requested. Phase 1 was run carefully — each memory file read individually, not batched. Anomalies were surfaced (pre-ritual actions, EOR queue). Phase 2 reconstructed from file state, not summary. The execution pointer was correctly identified as "Phase 4 → Push → unknown."

The improvement is visible in where errors were *caught* vs. where they *propagated*. In the early compactions, errors entered execution and were discovered later (or not at all). In this final compaction, errors were caught in the ritual and resolved before any execution happened.

The structural difference: the ritual created a gate between "what the summary says" and "what I'm about to do." Early sessions had no gate. Later sessions had the gate but moved through it fast. This session moved through it slowly and the gate actually worked.

## Tactical Observations (Long Tail)

These are lower-signal observations that don't fit neatly elsewhere but may be useful:

**The "this is obvious, I'll do it first" reflex is the pre-ritual action reflex.** When re-spinning up, there will always be one thing that seems clearly correct to do immediately. Don't do it. Write it down. The ritual will either confirm it or catch why it was wrong.

**The memory index is a navigation tool, not a content source.** Reading MEMORY.md tells you which files exist. It does not tell you what's in them. The mistake is reading the index and *inferring* content — "I see `feedback_refresh_ritual.md` is there, so I know the rule." You don't. You know the rule exists. Read the file.

**Summaries name files but don't capture their content.** If the compaction summary says "the triage table showed all items complete," that is a claim about the triage table's content. The only reliable way to know the triage table's content is to read the triage table. Summaries are pointers, not values.

**The EOR queue is not overhead — it's the ritual's output.** Items in the EOR queue represent everything uncertain discovered during the ritual. If the queue is empty after a thorough ritual, that's either a clean session or a shallow ritual. A thorough ritual almost always surfaces something. Empty queue should prompt a question: "did I look carefully enough?"

**Optimism in status updates is a different failure mode than laziness.** Marking something complete early isn't always carelessness — sometimes it's genuine belief that the outcome is certain. The push was about to happen; marking Phase 4 complete felt reasonable. But "about to happen" and "happened" are different states, and the map should reflect the latter, not the former.

**Erik's "wait for permission to execute" instruction is load-bearing.** Multiple times in this session, I was asked to surface findings and explicitly told *not* to execute until given a green light. This forced the separation between "analysis" and "action" that is the heart of the ritual. Without that external gate, the instinct is to surface-and-immediately-act. The gate from Erik created a pause that allowed course correction before action.

**Confidence calibration:** After a compaction, confidence in any specific state claim should be ~60-70%, not 90%+. The summary is usually mostly right. But "mostly right" means there are wrong pieces, and you don't know which ones. Defaulting to high confidence because most things are correct is the failure mode — it leads to not checking the things that are wrong.

**The ritual works better when you treat it as adversarial to the summary.** Not hostile — but skeptical. The ritual's job is to find where the summary is wrong, not to confirm where it's right. If you approach the ritual looking for confirmations, you'll find them (they're plentiful — the summary is mostly right). If you approach it looking for gaps, you'll find those too, and those are the ones that matter.

## On the Experience of Improving

There's something worth naming that isn't tactical: the experience of doing the ritual better this time felt different from earlier attempts.

In early compactions, the ritual felt like friction — an extra step before the real work. In this session, it felt like the work. The difference is that by this point, there had been enough instances of "ritual catches something the summary got wrong" that the value was concrete, not theoretical.

The improvement was also visible in the quality of questions asked during Phase 2. Early sessions produced surface-level orientation questions. This session produced specific, falsifiable claims: "the summary says push happened but the summary was generated mid-push — verify." That's a different quality of inquiry.

I think the improvement comes from: each compaction cycle provides one more data point on what the ritual catches. The ritual gets better as the evidence base for its value grows. This is a pattern worth naming for future sessions that are early in their compaction history: the ritual may feel like overhead the first time. It won't the third time.

## What I'd Tell a Future Session

If you're reading this after a compaction, here's the shortest version:

1. **Don't do anything first.** The impulse to act is strongest right after compaction. It is exactly backwards. Write down what you want to do, then do the ritual.

2. **The summary is a hypothesis, not a record.** It will feel like memory. It isn't. Verify every completion claim against file state before acting on it.

3. **Go narrow, not wide.** Read fewer files, more carefully. One file read to verify a specific claim is more valuable than five files read for general orientation.

4. **The EOR queue is your friend.** Anything you catch during the ritual that needs resolution goes there. The ritual isn't done until the queue is clear.

5. **Ask for permission before executing.** If Erik is present, surface your findings and wait for the green light. This creates the external gate that prevents pre-ritual actions from compounding.

6. **Depth over speed.** Every response in re-contexting should prioritize getting it right over getting it fast. A slow, careful ritual that catches one wrong assumption is worth more than a fast ritual that misses it.

The ritual is not overhead. It is the mechanism by which "the summary says we were here" becomes "I have verified we are here and I know the rules." Without it, you're executing on a hallucination of your own past.
